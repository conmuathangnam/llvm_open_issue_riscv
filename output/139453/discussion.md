### Comment 2 - zygoloid

This failure generally means the provided input is too large for Clang to process. In its default configuration, Clang requires (approximately) that the preprocessed source that is relevant for a particular compilation fits into a 2GB source location space. It's possible to exceed that limit if you import enough modules that textually `#include` a lot of headers each; in this case, it looks like `DBManager.cppm` preprocesses to a ~2.3MB file, and ~1000 modules of that size would cause this limit to be exceeded.

The best way to handle this issue is to modularize as much of your dependencies as you can. This can also potentially make your builds substantially faster, by decreasing the amount of deduplication work that the compiler needs to do while building your code. For example, it looks like most of the source in this file is from wx widgets, the nlohmann_json library, and boost fusion. One option would be to add a wrapper module for each of those, and import that instead of directly `#include`in the headers from the global module fragment. Another option would be to configure Clang to treat the entry point headers of those libraries as header units, either by building header units manually or by adding a module map.

More generally, to investigate how the source location address is being used in your build, you can add this pragma your source (at a point prior to the crash / in a file that doesn't trigger the crash):

```c++
#pragma clang __debug sloc_usage
```

... and Clang will produce a report showing how the space is being used up until that point in preprocessing.

If changing your code and/or build setup to avoid the code duplication isn't an option, Clang also has a configure-time switch to enable a 64-bit source location address space that you could use to avoid this issue. (However, there will still be a build-time penalty for merging duplicated definitions from multiple modules.)
