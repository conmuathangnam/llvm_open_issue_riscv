## **1\. Executive Summary**

An Internal Compiler Error (ICE) was identified in the LLVM WebAssembly backend during a new toolchain integration. The failure manifests as a "Cannot select" error during instruction selection for wasm32-unknown-emscripten targets when Wasm SIMD is enabled. The crash is triggered by vectorized comparisons of 128-bit objects.

## **2\. Incident Diagnosis**

Failure Log:

```
fatal error: error in backend: Cannot select: 0x12ab7fd824d0: i32 = setcc 0x12ab7fd82bd0, Constant:i1<0>, seteq:ch
  0x12ab7fd82bd0: i1 = truncate 0x12ab7fd82540
    0x12ab7fd82540: i32 = llvm.wasm.anytrue Constant:i32<15562>, 0x12ab7fd821c0
      0x12ab7fd821c0: v16i8,ch = load<(load (s128) from `ptr null`, align 1)> 0x12ab7fc9dd18, Constant:i32<0>, undef:i32
In function: _Z13trigger_crash9StringRef
```

### **Root Cause Analysis**

The failure stems from the creation of an illegal **i1** (1-bit integer) logical node late in the SelectionDAG pipeline. This node is generated by the global Store-To-Load Forwarding (STLF) optimization. In WebAssembly, i1 is an illegal register type; all boolean operations are natively represented as i32 registers. The backend lacks instruction selection patterns to lower scalar i1 comparisons derived from truncated SIMD reduction results.

## **3\. Culprit Identification**

The regression was introduced by commit [**a994198**](https://github.com/llvm/llvm-project/commit/a994198906cb4e0100f78c39db07f7b9f5410fc2) (*"Reland: Enable bitcast STLF for Constant/Undef"*). This change enabled Store-To-Load Forwarding for paths involving bitcasts and truncations. In the case of WebAssembly, it forces boolean reduction results through memory by creating a truncate to i1 node, which the backend cannot select.

## **5\. Proposed Resolution (3-Part Fix)**

A multi-layered fix is required to resolve the crash and prevent future regressions. TableGen patterns using i1 are illegal and must be avoided.

### **Part 1: Backend Safety Net (Mandatory)**

Modify performSETCCCombine in WebAssemblyISelLowering.cpp to catch and fold late-created i1 nodes back into i32 space. This must be placed **above** the isBeforeLegalize() check.  

```c++
if (LHS->getOpcode() == ISD::TRUNCATE && isNullConstant(RHS)) {
  SDValue S0 = LHS->getOperand(0);
  if (S0->getOpcode() == ISD::INTRINSIC_WO_CHAIN) {
    unsigned IntNo = S0->getConstantOperandVal(0);
    if (IntNo == Intrinsic::wasm_anytrue || IntNo == Intrinsic::wasm_alltrue) {
      return DCI.DAG.getSetCC(SDLoc(N), N->getValueType(0), S0,
                              DCI.DAG.getConstant(0, SDLoc(N), S0.getValueType()),
                              CC);
    }  
  }  
}
```

### **Part 2: Internal Logic Hygiene**

Update performAnyAllCombine in WebAssemblyISelLowering.cpp to perform comparisons directly in i32, avoiding the i1 detour in the backend's own optimization logic.

### **Part 3: ISel Mapping**

Add legal TableGen patterns to WebAssemblyInstrSIMD.td to map i32 zero-checks on SIMD reductions to efficient machine instructions (e.g., i32.eqz).  
`def : Pat<(i32 (seteq (i32 (intrinsic (vec.vt V128:$x))), (i32 0))), (i32 (EQZ_I32 (inst $x)))>;`

#183866 avoids this problem by only creating MVT::i32 nodes so it can run after legalization.
It's essentially "Part 2" of the LLVM suggestion above and it fixes the proximate cause of the crash.
(The bug was always there but the recent change exposed it by causing the DAG combine to trigger after legalization).
The "Part 1" suggestion would correct the problem in the setcc but wouldn't prevent the creation of illegal nodes and doesn't seem like it would cover all the possible cases.
