### Comment 4 - nikic

This issue is related to the use of `-ffat-lto-objects`.

Let's take a simple input:
```c++
#include <coroutine>

class BasicCoroutine {
public:
    struct Promise {
        BasicCoroutine get_return_object() { return BasicCoroutine {}; }

        void unhandled_exception() noexcept { }

        void return_void() noexcept { }

        std::suspend_never initial_suspend() noexcept { return {}; }
        std::suspend_never final_suspend() noexcept { return {}; }
    };
    using promise_type = Promise;
};

BasicCoroutine coro() {
    co_return;
}

int main() {
    coro();
}
```
Then `clang++ -std=c++23 -flto=thin -ffat-lto-objects -c -O3 test.cpp` is sufficient to reproduce the issue.

I believe the problem is that buildFatLTODefaultPipeline never actually schedules the coroutine passes for the "main" module (that gets lowered to ELF), because they're neither part of the pre-link ThinLTO pipeline, nor part of the module optimization pipeline.
